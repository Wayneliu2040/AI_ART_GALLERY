# 按周排期的开发时间表与团队分工表

## 规划假设

本排期默认以下前提成立：

- 团队成员为 3 人
- 项目开发周期为 8 周
- 目标是在 Azure 上完成可演示的 MVP 部署

如果实际团队只有 2 人，文档最后提供了简化版分工建议。

## 团队角色建议

### 成员 A：前端负责人

主要负责：

- React 项目初始化
- 页面和公共组件开发
- 前端 API 接入
- 登录状态管理和路由保护
- 画廊页、上传页、详情页的交互实现

### 成员 B：后端负责人

主要负责：

- .NET Web API 初始化
- 认证模块
- 图片、评论、点赞相关 API
- 业务逻辑实现
- DTO 和接口返回结构设计
- SQL 数据访问逻辑

### 成员 C：云平台与数据负责人

主要负责：

- Azure SQL 配置
- Azure Blob Storage 配置
- Azure VM 部署
- Azure Static Web Apps 部署
- 环境变量和配置管理
- 联调支持与部署问题排查

## 每周开发排期

## 第 1 周：项目启动与环境准备

### 目标

- 明确项目范围
- 初始化仓库和工程结构
- 搭建本地开发环境

### 任务安排

成员 A：

- 初始化 React 项目
- 搭建路由和页面骨架
- 整理前端目录结构

成员 B：

- 初始化 .NET Web API 项目
- 配置基础分层结构
- 启用 Swagger

成员 C：

- 准备 Azure 账号和资源规划
- 制定 Azure 资源命名规则
- 整理部署前提条件清单

### 本周交付物

- 前后端都能本地启动
- 仓库结构稳定
- Azure 准备清单完成

## 第 2 周：数据库与对象存储搭建

### 目标

- 完成数据模型设计
- 打通 Azure SQL 和 Azure Blob Storage

### 任务安排

成员 A：

- 根据数据模型细化页面和组件清单

成员 B：

- 定义实体类和 DTO
- 配置 EF Core
- 编写 Migration 或建表脚本

成员 C：

- 创建 Azure SQL Database
- 创建 Storage Account 和 Blob 容器
- 验证存储连接

### 本周交付物

- ER 图完成
- SQL 表结构完成
- Blob 容器可用

## 第 3 周：认证模块开发

### 目标

- 完成注册和登录流程

### 任务安排

成员 A：

- 开发登录页
- 开发注册页
- 实现前端登录状态管理

成员 B：

- 实现注册 API
- 实现登录 API
- 完成密码哈希和 token 生成

成员 C：

- 验证后端到 Azure SQL 的访问
- 协助管理配置项和密钥

### 本周交付物

- 用户可以注册
- 用户可以登录
- 受保护页面具备登录校验基础

## 第 4 周：图片上传模块开发

### 目标

- 完成图片上传的完整链路

### 任务安排

成员 A：

- 开发上传页与表单
- 增加图片选择和元数据输入项
- 显示提交结果反馈

成员 B：

- 实现 `POST /api/images`
- 增加上传文件校验
- 将元数据写入 SQL

成员 C：

- 打通后端与 Azure Blob Storage 的连接
- 验证图片上传与 URL 生成

### 本周交付物

- 从前端上传图片到 Blob 并写入 SQL 的流程可用

## 第 5 周：画廊页与图片详情页

### 目标

- 完成图片浏览和详情展示

### 任务安排

成员 A：

- 开发画廊页
- 开发图片详情页
- 接入图片列表和详情接口

成员 B：

- 实现 `GET /api/images`
- 实现 `GET /api/images/{id}`
- 区分列表数据和详情数据返回字段

成员 C：

- 验证图片 URL 的访问效果
- 协助准备演示用测试数据

### 本周交付物

- 用户可以浏览已上传图片
- 用户可以进入详情页查看完整信息

## 第 6 周：搜索、评论和点赞

### 目标

- 完成用户交互功能

### 任务安排

成员 A：

- 增加搜索栏和标签筛选
- 接入评论和点赞按钮交互

成员 B：

- 实现搜索接口
- 实现评论接口
- 实现点赞接口

成员 C：

- 验证评论和点赞的数据写入
- 支持联调测试

### 本周交付物

- 搜索可用
- 评论可用
- 点赞可用

## 第 7 周：Azure 部署与系统联调

### 目标

- 将完整系统部署到 Azure

### 任务安排

成员 A：

- 准备生产环境前端配置
- 验证前端部署后的 API 地址和访问行为

成员 B：

- 发布后端构建产物
- 协助后端在 Azure VM 上部署
- 修复部署相关后端问题

成员 C：

- 将前端部署到 Azure Static Web Apps
- 将后端部署到 Azure VM
- 配置 SQL 和 Blob 的生产环境连接
- 验证公网访问

### 本周交付物

- 前端公网地址可访问
- 后端公网地址可访问
- 系统云端端到端链路可用

## 第 8 周：测试、修复与答辩准备

### 目标

- 稳定系统
- 完成答辩资料

### 任务安排

成员 A：

- 完善 UI 状态
- 修复前端缺陷
- 协助准备演示页面截图

成员 B：

- 修复 API 缺陷
- 完善错误处理
- 协助整理 API 文档

成员 C：

- 整理部署文档
- 整理 Azure 架构说明
- 进行最终部署验证

### 本周交付物

- 稳定可演示的 MVP
- 完整的展示材料
- 完整的文档资料

## 每周检查建议

每周结束时建议固定检查以下内容：

- 本周已完成内容
- 当前阻塞问题
- 下周目标
- GitHub 合并情况
- Azure 配置风险

建议每周例会固定回答以下问题：

1. 本周完成了什么
2. 当前卡住了什么
3. 当前开发是否仍然围绕 MVP
4. 是否有 Azure 配置问题需要提前处理

## 分工总结

### 成员 A

- 前端页面
- 前端组件
- 前端 API 接入
- 页面展示与交互逻辑

### 成员 B

- 后端 API
- 认证逻辑
- 业务逻辑
- SQL 数据访问

### 成员 C

- Azure 服务
- 对象存储接入
- 部署
- 环境配置与联调支持

## 如果团队只有 2 人

可以按以下方式合并分工：

### 成员 A

- 负责全部前端工作
- 负责部分联调测试
- 负责部分文档整理

### 成员 B

- 负责后端 API
- 负责 Azure SQL
- 负责 Azure Blob Storage
- 负责 Azure 部署
- 负责部分文档整理

此时建议进一步控制范围，只保留最严格的 MVP：

- 注册
- 登录
- 上传
- 浏览
- 搜索
- 评论或点赞，至少有一个完全做完

## 最后建议

每周推进时最重要的是：

1. 尽早打通一条完整业务链路
2. 尽早验证 Azure 接入，而不是留到最后
3. 预留足够时间修 bug 和准备答辩

对于学生项目来说，一个规模更小但真正能跑通、能部署、能解释清楚的系统，通常比一个功能很多但完成度不足的系统更有说服力。
